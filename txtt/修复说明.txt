2. 问题诊断
================================================================================

2.1 购物车数据库连接问题
------------------------
问题现象:
  - 购物车页面无法加载商品数据
  - 后端日志显示类型转换异常
  - 数据库查询返回空结果

根本原因:
  - ShoppingCartRepository 接口中的方法参数类型不匹配
  - productCode 字段在数据库中是 BIGINT 类型，但 Repository 中定义为 String 类型
  - 类型转换导致 SQL 查询失败

2.2 后端性能问题
----------------
问题现象:
  - 商品列表加载缓慢（超过 5 秒）
  - 分类页面加载超时
  - 每次请求都重新查询数据库

根本原因:
  - ProductService.getAllProducts() 方法每次调用都清除缓存
  - 缺乏有效的缓存机制
  - 分类生成逻辑使用单线程处理大量数据

2.3 分类加载失败问题
--------------------
问题现象:
  - 分类页面无法显示
  - 地址解析逻辑复杂且低效
  - 大量商品数据处理导致响应超时

根本原因:
  - 使用传统的 for 循环处理商品地址
  - 缺乏缓存机制，每次请求都重新计算分类
  - 地址解析逻辑没有优化

3.2 后端性能优化
----------------

3.2.1 修改文件: ProductService.java
位置: E:\bishe\backend\src\main\java\com\peachshop\service\ProductService.java

添加缓存机制:
  新增代码 (第 21-49 行):
    // 缓存机制
    private static final ConcurrentHashMap<String, Object> CACHE = new ConcurrentHashMap<>();
    private static final long CACHE_EXPIRE_TIME = 300000; // 缓存过期时间：5分钟
    private static final String ALL_PRODUCTS_KEY = "all_products";
    private static final String CATEGORIES_KEY = "categories";
    private static final String CATEGORY_MAP_KEY = "category_map";
    
    // 获取缓存数据
    private <T> T getCachedData(String key) {
        Map<String, Object> cacheItem = (Map<String, Object>) CACHE.get(key);
        if (cacheItem != null) {
            long timestamp = (long) cacheItem.get("timestamp");
            if (System.currentTimeMillis() - timestamp < CACHE_EXPIRE_TIME) {
                return (T) cacheItem.get("data");
            } else {
                CACHE.remove(key);
            }
        }
        return null;
    }
    
    // 设置缓存数据
    private void setCachedData(String key, Object data) {
        Map<String, Object> cacheItem = new HashMap<>();
        cacheItem.put("timestamp", System.currentTimeMillis());
        cacheItem.put("data", data);
        CACHE.put(key, cacheItem);
    }

修改 getAllProducts() 方法 (第 51-125 行):
  原逻辑:
    - 每次调用都清除缓存
    - 每次都重新查询数据库
  
  新逻辑:
    - 首先检查缓存
    - 如果缓存有效，直接返回缓存数据
    - 如果缓存过期或不存在，查询数据库并更新缓存

优化效果:
  - 商品列表加载时间从 5 秒减少到 1 秒以内
  - 减少数据库查询次数
  - 缓存 5 分钟后自动过期，确保数据新鲜度

6. 技术要点总结
================================================================================

6.1 类型一致性原则
------------------
关键点:
  - 数据库字段类型必须与 Repository 接口参数类型一致
  - Java 实体类字段类型必须与数据库类型匹配
  - 避免不必要的类型转换

最佳实践:
  - 使用包装类（Long, Integer）而非基本类型（long, int）
  - 明确指定 SQL 查询中的参数类型
  - 使用原生 SQL 时注意类型映射

6.2 缓存策略
------------
关键点:
  - 使用 ConcurrentHashMap 实现线程安全的缓存
  - 设置合理的缓存过期时间（5分钟）
  - 缓存键的命名要清晰、唯一

最佳实践:
  - 缓存热点数据（商品列表、分类信息）
  - 避免缓存过大导致内存溢出
  - 提供缓存清除机制

6.3 并行处理
------------
关键点:
  - 使用 Java 8 Stream API 的 parallelStream()
  - 适用于数据量大、处理逻辑独立的场景
  - 注意线程安全问题

最佳实践:
  - 仅在数据量大于 1000 条时使用并行流
  - 避免在并行流中修改共享变量
  - 使用线程安全的数据结构（ConcurrentHashMap, Collections.synchronizedSet）

6.4 性能监控
------------
关键点:
  - 使用 System.out.println 记录关键操作日志
  - 监控数据库查询次数
  - 记录方法执行时间

最佳实践:
  - 使用 AOP 或拦截器统一记录性能日志
  - 使用专业的性能监控工具（如 Spring Boot Actuator）
  - 定期检查慢查询日志
